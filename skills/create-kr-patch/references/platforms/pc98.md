# PC-98 (NEC PC-9801/9821) 한글 패치 참조

## 1. 플랫폼 개요

NEC PC-9801은 1982년 출시된 일본 국내 표준 16비트 PC 라인업이다. IBM PC와 호환되지 않는 자체 아키텍처이며, CPU는 8086 또는 NEC V30(8086 슈퍼셋, 80186 명령어 포함)이고 후기 모델은 80286/80386을 탑재한다. 1990년 전후 게임은 모델 호환성을 위해 공통분모인 8086 명령어 셋으로 빌드된 경우가 많다. 디스어셈블 중 80186+ 명령어(`PUSHA`, 즉시값 시프트 등)가 보이면 V30 이상 전용 빌드라는 단서다. 패치 코드도 8086 호환을 유지하려면 V30 고유 명령어를 쓰지 않는다.

화면은 텍스트 plane과 그래픽 plane이 분리된 dual-GDC(µPD7220 ×2) 구조다. master GDC가 텍스트 plane(80×25 문자 그리드, 시스템 한자 ROM 합성), slave GDC가 그래픽 plane(640×400, 4 plane = 16색)을 담당하고, 텍스트 plane이 그래픽 plane 위에 오버레이된다. 게임 텍스트가 어느 plane에 그려지는지 식별하는 것이 텍스트·폰트 분석의 첫 단계다.

콘솔과 달리, 코드가 ROM이 아니라 **디스크에서 RAM으로 적재되어 실행**된다. 패치 대상은 ROM 바이너리가 아니라 디스크 이미지 안의 파일/섹터 데이터이고, 디스어셈블은 어떤 파일이 어떤 메모리 주소에 적재되는지부터 알아내야 한다.

## 2. 메모리 모델

8086 리얼 모드는 모든 주소를 세그먼트:오프셋 쌍으로 표현한다. 물리 주소 = `SEG × 16 + OFF`(20비트, 1MB 공간). flat 주소 공간이 아니므로 같은 물리 주소를 여러 (seg, off) 조합으로 표현할 수 있다.

| 세그먼트:오프셋 | 물리 주소 | 용도 |
|---|---|---|
| `0000:0000`– | `0x00000`– | 인터럽트 벡터 테이블 (256 × 4B) |
| `0060:0000`–`9FFF:000F` | `0x00600`–`0x9FFFF` | 메인 RAM (~640KB conventional) |
| `A000:0000`– | `0xA0000`–`0xA3FFF` | 텍스트 VRAM (TVRAM) |
| `A400:0000`– | `0xA4000`– | CG 윈도우 (한자 ROM 액세스) |
| `A800:0000`– | `0xA8000`– | 그래픽 VRAM plane 0 (B) 32KB |
| `B000:0000`– | `0xB0000`– | 그래픽 VRAM plane 1 (R) 32KB |
| `B800:0000`– | `0xB8000`– | 그래픽 VRAM plane 2 (G) 32KB |
| `E000:0000`– | `0xE0000`– | 그래픽 VRAM plane 3 (I) 32KB |
| `F800:0000`– | `0xF8000`– | BIOS / IPL ROM |

패치 코드 작성 시 주의점:

- near `JMP`/`CALL`은 16비트 displacement로 같은 CS 안 ±32KB까지만 직접 도달한다. 더 멀면 `JMP FAR seg:off`(5바이트)가 필요하므로 트램폴린의 점프 거리를 항상 확인한다.
- 조건 분기는 8비트 displacement(-128~+127)만 허용한다.
- 텍스트 처리 루틴은 거의 항상 `LODSB`/`STOSB`/`MOVSB` string 명령으로 짜이므로, 문자열 디코더를 찾는 단서가 된다.
- `OUT DX, AL` 시퀀스 주변은 GDC/FDC/사운드 칩 제어 코드다.

자주 쓰는 BIOS 인터럽트: `INT 18h`(CRT/한자 표시), `INT 1Bh`(디스크 read/write), `INT 21h`(DOS API). 디스크→RAM 적재 매핑은 `INT 1Bh` 호출 인자(트랙·섹터·길이·대상 주소)에서 직접 읽어낸다.

## 3. 그래픽

그래픽 VRAM은 32KB × 4 plane = 128KB의 planar 구성이다. 각 plane이 픽셀당 1비트를 담당하고, 4개 plane의 같은 비트 위치가 합쳐져 4비트 색 인덱스(16색)가 된다.

```
픽셀 N의 색 = (B비트N) | (R비트N << 1) | (G비트N << 2) | (I비트N << 3)
```

폰트 비트맵도 이 구조를 따른다. 글자가 단색이면 plane 한 개에만 데이터가 있는 1bpp이고, 컬러 글자면 4 plane 모두에 비트맵이 있다.

| 용도 | 크기 | 1글리프 데이터량 (1bpp) |
|---|---|---|
| 전각(한자) | 16×16 | 32바이트 (16행 × 2바이트) |
| ANK 반각 | 8×16 | 16바이트 |

식별법: 후보 데이터의 첫 32바이트를 1bpp 16×16으로 시각화해 글자 모양이 보이면 1bpp planar 폰트다. 한글은 정보량 때문에 16×16이 표준이며, PC-98 게임 자체 폰트가 16×16 격자라면 한글 글리프를 같은 포맷으로 그대로 주입할 수 있다.

텍스트 출력 경로는 셋 중 하나(또는 혼합)다.

1. **텍스트 plane(TVRAM)** — SJIS 코드를 쓰면 시스템 한자 ROM이 글리프를 합성. 그래픽이 풍부한 RPG는 거의 쓰지 않는다.
2. **자체 폰트 비트맵을 그래픽 plane에 직접 그리기** — RPG/AVG 표준. 한글 패치는 이 폰트 데이터를 교체한다.
3. **한자 ROM 글리프를 fetch해 그래픽 plane에 복사** — `INT 18h` 또는 CG 윈도우(`0xA4000`대) 액세스가 단서. 이 경우 폰트가 디스크에 없고, PC-98 한자 ROM에는 한글이 없으므로 글리프 출력 경로 전체를 우회(한자 ROM 호출을 자체 한글 시트 lookup으로 hijack)해야 한다.

## 4. 디스크 포맷

### 4.1 D88 컨테이너

D88은 물리 디스크의 트랙·섹터 구조를 보존하는 컨테이너다. 모든 정수는 little-endian.

디스크 헤더(선두 0x2A0 또는 0x2B0 바이트):

| 오프셋 | 크기 | 필드 |
|---|---|---|
| `0x000` | 16 | 디스크 이름 (`0x00` 종결) |
| `0x01A` | 1 | Write protect (`0x10`=보호) |
| `0x01B` | 1 | Media type (`0x20`=2HD 1.2MB) |
| `0x01C` | 4 | 디스크 총 크기 (헤더 포함) |
| `0x020` | 4×164 | 트랙 오프셋 표 (`0`=미포맷), 인덱스 = `cylinder × 2 + head` |

트랙 오프셋 표는 160개(헤더 0x2A0) 또는 164개(0x2B0) 두 변형이 있다. 파서는 첫 트랙 오프셋 값이 0x2A0인지 0x2B0인지 보고 표 길이를 판단한다.

각 섹터는 16바이트 헤더 + 데이터다: C(cylinder)/H(head)/R(sector ID)/N(size code, 실제 크기 = `128 << N`; 2HD 표준은 N=3 → 1024B), 트랙당 섹터 수(WORD), density, DDAM, FDC status, 실데이터 길이(WORD, 오프셋 0x0E). 보호 디스크는 N과 실데이터 길이를 일부러 다르게 두기도 하므로 두 필드 모두 그대로 보존한다.

### 4.2 HDM (raw 포맷)

HDM은 헤더가 전혀 없는 NEC raw 이미지다. 1024B 섹터가 트랙 0 헤드 0 섹터 1부터 평탄하게 연결된다. 표준 2HD는 80 cylinder × 2 head × 8 sector × 1024B = 1,310,720B이며 파일 크기가 정확히 이 값이면 표준 포맷이다. HDM→D88은 섹터 헤더를 일률 생성(N=3, density=0x00)하면 무손실이고, D88→HDM은 비표준 정보(혼합 섹터 크기, 보호 패턴)가 있으면 손실된다. 게임이 HDM으로 덤프되어 유통된다는 사실 자체가 카피 프로텍션 없는 표준 2HD라는 방증이다.

### 4.3 재빌드 전 필수 검증

디스크 이미지를 읽고 다시 쓴 결과가 원본과 **byte-identical**해야 한다(무손실 round-trip). 헤더의 디스크명·reserved·write protect, 트랙 오프셋 표의 0 위치, 섹터 헤더 전 필드, 트랙 내 섹터 순서(인터리빙), 트랙 데이터 영역의 물리 배치 순서를 모두 보존한다. 대상 디스크 전수가 이 테스트를 통과한 뒤 다음 단계로 넘어간다.

### 4.4 FAT12 파일시스템

PC-98 부트 마커는 IBM PC의 `0x55 0xAA`와 다르다. 첫 섹터가 `EB ?? 90` 점프 + OEM 이름 + BPB로 시작하면 DOS FAT 디스크다. PC-98 FAT12의 함정은 **섹터당 1024바이트**(PC의 512가 아님)이며, 클러스터는 보통 1섹터다. BPB 핵심 필드: bytes per sector(`0x0B`), sectors per cluster(`0x0D`), number of FATs(`0x10`), root directory entries(`0x11`), sectors per FAT(`0x16`). 디렉토리 엔트리는 32바이트(파일명 8+3, 속성, first cluster `0x1A`, 크기 `0x1C`)이고 FAT 엔트리는 12비트가 3바이트에 2개씩 packed된다. 일부 게임 개발사는 자체 DOS 호환 미니 OS(예: Compile의 MEGDOS)로 부팅하는데, 파일시스템은 그대로 FAT12이므로 표준 FAT12 파서로 파일 추출이 가능하다.

## 5. 텍스트 엔진 패턴

PC-98 게임 텍스트는 거의 예외 없이 Shift_JIS다. 시스템 한자 ROM이 SJIS로 직접 인덱싱되고 동시대 도구체인이 SJIS를 가정하기 때문이다. SJIS 2바이트 문자의 1번째 바이트는 `0x81`–`0x9F` 또는 `0xE0`–`0xFC`, 2번째 바이트는 `0x40`–`0x7E` 또는 `0x80`–`0xFC`다.

스크립트 파일 구조 예:

```
0x0000-0x0220 : 헤더
0x0220-0x0620 : 포인터 테이블 (512 엔트리 × 2바이트 = 1KB)
0x0620-0x4910 : 텍스트 영역 1 — 메뉴/UI/회복/아이템 (~15,600바이트)
0x4910-       : 텍스트 영역 2 — 이벤트 dialog (포인터 테이블이 가리키는 영역)
```

포인터 테이블 식별법: 텍스트 영역 직전에 16비트 워드들이 군집해 있고, 값이 모두 텍스트 영역 범위 안에 들어가면(예: 512개 전부 0x4910–0x7C85 범위) 포인터 표다. 포인터 검증 없이 워드 군집 전체를 표로 오인하면 텍스트 영역을 포인터로 잘못 읽으므로, "전 엔트리가 valid 범위" 조건으로 표의 끝을 끊는다.

제어코드는 1바이트 코드가 아니라 **ASCII `$` + 1글자의 `$X` 패밀리**일 수 있다:

| 코드 | 의미 |
|---|---|
| `$e` | Wait (입력 대기) |
| `$r` | Newline (줄바꿈) |
| `$c` | PageClear (페이지 지움) |
| `$(` / `$)` | 따옴표 열기/닫기 |
| `$` + `0x00` | End (메시지 종결) |

같은 엔진 가족의 콘솔판이 `0xF8`(변수), `0xFC`(WAIT), `0xFD`(줄바꿈), `0xFE`(END), `0xFF`(PAGE) 같은 1바이트 제어코드를 쓰는 것과 대비된다. PC-98 원전이 ASCII 제어코드를 쓰고 콘솔 이식판이 1바이트 코드로 변형된 사례가 실재하므로, 양쪽 패턴을 모두 검색한다. 메시지 분류(대사/메뉴/시스템)에는 "제어코드 1개 이상 포함 = dialog" 같은 휴리스틱이 유효하다.

## 6. 한글 인코딩 설계 — free prefix 탐색법

게임이 SJIS를 그대로 쓴다면 한글 음절은 SJIS의 **미사용 lead byte(free prefix)** 에 2바이트로 끼워 넣는다. trail은 SJIS 규칙 그대로 `0x40`–`0x7E`(63개) + `0x80`–`0xFC`(125개) = **prefix당 188 값**을 쓴다.

```
prefix #1 + trail = 음절 #0–187
prefix #2 + trail = 음절 #188–375
prefix #3 + trail = 음절 #376–563
→ N prefix × 188 = (N×188) 음절
```

음절 상한은 「확보한 free prefix 수 × trail 188」 공식의 출력이다. 전 코퍼스 기준 0회인 prefix가 몇 개냐에 따라 상한이 정해지고, prefix를 더 확보하면 188 단위로 늘어난다(글리프 예산 산정의 일반 공식은 `references/strategy/font-strategy.md` 3절).

### 6.1 가설을 글리프 통계로 검증해 정정한 사례

free prefix 선정은 반드시 **전체 코퍼스 합산 통계**로 검증해야 한다. 한 사례에서 일어난 정정 과정:

1. **최초 가설**: 주 스크립트 파일 하나의 lead byte 분포만 sweep한 결과 `0xEB`–`0xFC` 구간이 전부 0회 → 디코더의 range 검사가 단순해지는 연속 3바이트 `0xFA`/`0xFB`/`0xFC`를 prefix로 채택했다.
2. **글리프 전수 통계로 반증**: 전 디스크 + 전 실행 파일을 합산해 SJIS pair 단위로 집계하자 `0xFA`/`0xFB`/`0xFC`가 모두 **cp932 외자(gaiji) 글리프**로 실사용 중이었다. 게임은 외자 영역(lead `0xED` 이후)에서 외자 글리프를 캐릭터 마크/UI 심볼 등으로 적극 쓰고 있었고, 한글 prefix가 이 영역을 잡으면 외자 글리프가 한글로 오해석되어 게임이 깨진다.
3. **정정**: 전 코퍼스에서 진짜 0회인 lead는 `0xEB`/`0xEC`/`0xEF` 셋뿐 → 이를 최종 prefix로 채택했다. prefix 수가 같으므로 음절 상한은 동일하게 유지된다. 단 연속 범위가 아니므로 디코더 분기가 단순 range 검사보다 약간 복잡해진다.

단일 파일 분포만 보면 false negative가 생긴다. prefix 후보는 **모든 디스크의 모든 텍스트·코드 영역을 합산한 글리프 사용 통계**로 0회임을 확인한 뒤에 확정한다.

### 6.2 산정 결과의 운용

확보한 prefix 수로 정해지는 음절 상한은 KS X 1001 표준 한글 2,350자의 일부에 그칠 수 있다. 이 경우 번역 코퍼스의 음절 빈도를 집계해 상한 안에서 빈도순으로 동적 매핑하고, 부족하면 번역 단계에서 해당 음절을 피해 윤문하거나 자모 분해 출력으로 확장한다. 커버리지는 게임 분량과 문체에 따라 달라지므로 게임마다 다시 산정한다.

## 7. 재빌드 정책

번역 텍스트를 디스크에 다시 써넣는 정책은 세 가지다.

| 정책 | 방법 | 트레이드오프 |
|---|---|---|
| (a) 길이 보존 | 각 메시지를 원본 바이트 길이 안에 맞춤. 길면 자르거나 `$r`로 분할 | 포인터 표를 건드리지 않아 안전. 번역 표현력 제약 |
| (b) 포인터 재배치 | 텍스트 영역을 통째로 재배열하고 포인터 표 전 엔트리 재계산 | 길이 자유. 포인터 표 + 텍스트 영역 양쪽 갱신 필요, 접미사 공유 최적화가 있으면 보존/해제 결정 필요. 파일 크기가 늘면 FAT 클러스터 체인까지 영향 |
| (c) 하이브리드 | 고정 폭 UI(메뉴/아이템명)는 (a), 가변 길이 dialog는 (b) | 영역별 위험·이득에 맞춘 절충. 실전 권장 |

한국어는 일본어보다 같은 내용의 바이트 수가 다른 경우가 많아 순수 (a)는 빠듯할 수 있다. 코드가 길이를 하드코딩해 참조하는 메뉴/UI 영역은 (a)로 묶고, 포인터 테이블이 명시적으로 가리키는 dialog 영역은 (b)로 푸는 (c)를 우선 검토한다. 어느 정책이든 재빌드 후 디스크 이미지가 §4.3의 round-trip 인프라 위에서 생성되어야 한다.

## 8. 검증

| 환경 | 역할 |
|---|---|
| **np2debug** (Neko Project II 디버그 빌드) | 1차 분석 환경. 단계실행, 코드/메모리 브레이크포인트, ASM 뷰어, SJIS 메모리 뷰어, 스택 뷰어 |
| **MAME** (`pc9801` 계열 드라이버) | 자동화. 디버거 + Lua 스크립팅 + 트레이스 로그로 회귀 테스트 |
| **Neko Project 21/W** | 최종 QA. 일반 사용자 환경에서 패치 동작 확인 |

np2debug 표준 절차:

1. **출력 함수 식별** — 메시지 윈도우가 열리기 직전에 일시정지, 그래픽 VRAM(`0xA8000`)을 관찰하며 프레임 전진, 변화 직후의 CS:IP에서 호출 스택을 거슬러 함수 진입점을 찾는다.
2. **텍스트 RAM 위치 식별** — 화면 첫 글자의 SJIS 바이트(예: "ま" = `0x82 0xDC`)를 RAM 전체에서 검색, 다음 글자로 후보를 좁힌다.
3. **디스크→RAM 매핑** — 그 주소가 부팅 직후 비어 있으면 `INT 1Bh`에 브레이크포인트를 걸어 어느 트랙·섹터가 어느 주소로 적재되는지 표를 만든다.

패치 빌드 후 자동 회귀: 미리 녹화한 입력 시퀀스로 진행 → 정해진 프레임의 VRAM 캡처 → 기준 스크린샷과 비교해 한글 글리프 렌더링 깨짐을 잡는다. MAME 실행 예:

```bash
mame -debug pc9801vm -flop1 patched_system.d88 -flop2 sampling.d88 -flop3 data.d88
```

### 실기 구동 경로

수단은 세 갈래다.

- **플로피 이미지 → 실디스크**: 이미지를 실제 플로피로 굽는다. 노후 플로피·드라이브라 신뢰성이 낮다.
- **GOTEK + FlashFloppy**: FDI·HDM을 직접 지원한다. raw IMG는 `host=pc98` 설정이 필요하다(D88 직접 지원 여부는 확인 필요).
- **HDD 대체**: CF→IDE 변환, SCSI 경로는 BlueSCSI 등으로 대체한다.

제약 — IDE BIOS의 용량 한계가 있다(A-Mate 등은 약 540MB, 후기 기종은 4.3GB까지 — 모델별 확인 필요). GOTEK 핀아웃은 기종별로 다르고, 실기 입수성 자체가 어렵다.

**빌드 포맷에의 영향**: **이미지 포맷이 GOTEK 호환선(FDI/HDM)에 맞아야** 실기 변환이 매끄럽다. HDM/FDI를 1차 배포 포맷으로 두면 실기가 직접 인식하지만, D88 위주로 내면 변환이 한 단계 더 든다. HDD 설치형은 HDI로 만들어 SCSI·CF로 배포한다(7절 재빌드 정책에서 배포 포맷을 이 호환선에 맞춰 정한다).

함정 — 포맷 다양성(FDI/HDM/D88/NHD/HDI 등)이 이 플랫폼의 큰 함정이다. 어느 포맷이 어느 솔루션에서 직접 인식되는지 명시하지 않으면 사용자가 막힌다. IDE 540MB 벽도 주의한다. 실기 검증 기기 확보 난이도가 8종 중 가장 높은 데다, MiSTer PC-98 코어가 미완성이라 에뮬레이터와 실기 사이를 메울 성숙한 FPGA 중간 검증 수단이 사실상 없어, MAME 계열 정확 에뮬레이터가 검증의 중심이 된다.

## 9. 사례 요약 — PC-98 마도물어 1·2·3 합본

- **대상**: 합본 재발매판 7장(HDM, 게임별 System+Sampling 쌍 + 공유 Data 1장) + 1편 단독판 4장(D88, 영문 팬 패치 포함). 모두 표준 2HD 1.2MB.
- **파일시스템**: Compile MEGDOS(DOS 호환 미니 OS) 부팅 + FAT12. 표준 FAT12 파서로 파일 추출 가능.
- **폰트**: 게임 자체 폰트 파일(16×16 1bpp 격자) + 시스템 한자 ROM 병용. 한글은 16px 비트맵 폰트를 같은 격자로 변환해 주입.
- **텍스트**: SJIS. 스크립트 파일은 헤더 + 512 엔트리 포인터 테이블(0x0220–0x0620) + 텍스트 영역 2개(0x0620–0x4910 메뉴/UI, 0x4910+ 이벤트 dialog). 제어코드는 ASCII `$X` 패밀리(`$e` Wait, `$r` Newline, `$c` PageClear, `$(`/`$)` 따옴표, `$`+`0x00` End).
- **한글 인코딩**: free prefix `0xEB`/`0xEC`/`0xEF` × trail 188. 최초 가설 `0xFA`–`0xFC`는 글리프 전수 통계에서 외자 글리프 점유가 확인되어 폐기.
- **분석 지렛대**: 1편 단독판의 일어판↔영문 패치판 diff가 텍스트 인코딩·엔진의 일차 단서. 영문 패치가 System disk만 변경했다는 사실에서 텍스트가 System disk에 집중됨을 역산.
- **검증**: HDM/D88 round-trip byte-identical 확인 후 np2debug에서 부팅·텍스트 표시 검증, 동일 디스크의 복수 덤프(alt dump) 간 일치 비교.
