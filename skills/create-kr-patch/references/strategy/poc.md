# PoC — 한글 1자 화면 출력

## 1. PoC의 목표와 범위

한글 패치의 가장 큰 리스크는 번역이 아니라 "이 게임에서 한글이 화면에 나오기는 하는가"다. PoC(개념 증명)는 이 가설을 최소 비용으로 검증하는 단계이며, 목표는 단 하나 — **한글 1자를 실제 게임 화면에 출력**하는 것이다.

"한글 1자 출력"이 사소해 보여도, 이 한 글자가 통과하면 본 구현의 3대 전제가 동시에 검증된다.

| 검증 요소 | 질문 | 실패하면 |
|-----------|------|----------|
| **글리프 공간** | 한글 글리프 비트맵을 둘 자리가 있는가 — 폰트 시트의 교체 가능 슬롯, ROM/디스크의 빈 영역, VRAM 타일 한도 | 글리프를 아무리 잘 그려도 넣을 곳이 없다 |
| **인코딩 공간** | 텍스트 데이터 안에서 한글을 가리킬 코드 포인트가 있는가 — 미사용 문자 코드, 교체 가능한 기존 코드, 확장 prefix 영역 | 글리프가 있어도 스크립트에서 참조할 방법이 없다 |
| **렌더링 경로** | 코드 → 글리프 fetch → (변환) → 화면 픽셀의 전체 경로가 한글 글리프에도 그대로 동작하는가 | 글리프와 코드가 있어도 화면에 도달하지 못한다 |

세 요소는 서로 독립적으로 실패할 수 있다. 글리프 공간이 충분해도 인코딩이 기존 코드와 충돌할 수 있고, 둘 다 확보해도 렌더링 경로가 특정 코드 영역만 폰트로 처리하면 출력되지 않는다. 한글 1자 출력은 세 요소를 한꺼번에 점검하는 최소 단위 테스트다.

**범위 제한** — PoC는 다음을 다루지 않는다. 끌어들이는 순간 PoC가 아니라 본 구현이 된다.

- 전체 텍스트 번역, 번역 품질
- 가변폭 렌더링, 자동 줄바꿈
- 전체 글리프 서브셋 수용 (수백~수천 자)
- 포인터 전량 수정, 디스크 재배치 같은 빌드 인프라

단, PoC를 설계하면서 위 항목들의 *상한*은 같이 가늠해 둔다. 예컨대 새턴 마도물어 사례에서는 타일맵 엔트리의 비트 폭으로부터 글리프 슬롯 상한이 산정되었고, 이 수치가 서브셋 전략 전체를 규정했다. 플랫폼별 타일·VRAM 한도 산정법은 `references/platforms/` 아래 해당 플랫폼 문서를 본다.

## 2. 최소 경로 패턴 — 난이도 순 3단계

PoC 경로는 "기존 시스템을 얼마나 건드리는가"로 난이도가 갈린다. 낮은 단계부터 검토하되, 각 단계가 검증하는 범위가 다르다는 점을 기록한다. 최종 PoC 통과 선언은 실제 게임 데이터 경로에서 글자 코드가 글리프로 해석되어 화면에 도달했을 때만 한다.

### 단계 1 — 기존 글리프 1자 교체 (코드 변경 0)

게임이 폰트를 ROM/디스크 안에 데이터로 갖고 있다면, 폰트 시트의 글리프 1개를 한글 비트맵으로 덮어쓰는 것이 최소 경로다. 코드 패치가 전혀 없으므로 검증되는 것은 순수하게 "글리프 데이터 포맷을 올바르게 이해했는가"와 "렌더링 경로가 글리프 내용에 무관하게 동작하는가"다.

절차:

1. 폰트 데이터의 위치와 포맷(타일 크기, bpp, 비트플레인 배열, 압축 여부)을 확정한다.
2. 게임 내 자주 등장하는 글자(히라가나 등) 1개의 글리프를 한글 비트맵으로 교체한다.
3. 압축 폰트라면 재압축까지 포함해 다시 빌드한다.
4. 게임을 부팅해 해당 글자가 나오는 장면에서 한글이 표시되는지 확인한다.

압축 폰트가 있는 게임에서는 이 패턴이 "압축 해제 → 글리프 패치 → 재압축 → 원래 컨테이너 기록" 전체를 검증하는 최소 절차가 된다. 이 단계에서는 인코딩 공간 검증이 빠져 있다는 점에 주의한다(기존 코드를 그대로 쓰므로). 따라서 이 단계만으로는 "글리프/렌더 포맷 PoC"이며, 최종 PoC로 승격하려면 스크립트의 텍스트 바이트를 교체한 글리프의 코드로 바꿔 실제 텍스트 경로까지 통과시킨다.

### 단계 2 — BIOS/내장 폰트 훅 (후킹 1곳)

폰트가 게임 데이터 밖에 있으면 — 시스템 BIOS의 내장 폰트를 호출해 글리프를 받아오는 구조 — 글리프 교체가 원천적으로 불가능하다. 이때는 **글리프 fetch 호출 지점 1곳을 후킹**해, 특정 코드 영역을 만나면 BIOS 대신 자체 임베드한 한글 글리프를 반환하게 한다.

핵심 설계 원칙: **fetch만 가로채고, 그 이후의 변환·전송 경로는 원본 코드를 그대로 태운다.** 훅 함수는 원본 fetch 루틴과 동일한 입출력 규약(입력 레지스터, 출력 버퍼, 글리프 바이트 포맷)을 지켜야 하며, 자체 인코딩 영역 밖의 코드는 원본 루틴으로 그대로 넘긴다.

시스템 폰트 의존 게임에서는 글리프가 BIOS나 폰트 ROM 안에 있어 직접 교체가 불가능할 수 있다. 이때 텍스트 인터프리터가 글리프 fetch 함수를 호출하는 지점을 찾고, 특정 코드 영역만 자체 글리프 공급 함수로 우회시킨다. 자체 함수는 원본 fetch 출력과 동일한 포맷으로 출력 버퍼를 채우고, 한글 코드가 아니면 원본 fetch 루틴으로 되돌린다. 이후의 비트 심도 변환과 VRAM 전송은 원본 코드가 그대로 처리하므로, 검증해야 할 핵심은 호출 지점 수, 입출력 규약, 보존해야 할 레지스터, 자체 글리프 데이터 위치다.

### 단계 3 — 자체 디스패치 (텍스트 처리 경로 신설)

글리프 교체도, fetch 후킹도 답이 안 되는 경우가 있다.

- 타일이 너무 작아(8x8 등) 한글 완성형을 1글리프로 표현할 수 없고, 자모를 실행 시점에 합성해야 하는 경우
- 텍스트 엔진의 literal 처리 자체가 한글 코드 체계와 양립 불가능한 경우

이때는 텍스트 엔진의 문자 처리 분기에서 자체 렌더링 루틴으로 디스패치하는 경로를 새로 작성해야 한다. GG 마도물어 1 사례가 이 부류다 — 영문 패치의 가변폭 폰트(VWF) 엔진은 인접 글리프를 8x8 타일 경계에 걸쳐 비트 OR로 합성해 채워 넣는 구조라 2바이트 한글 코드·1글자 1타일 출력과 양립하지 않았고, 한글 프리픽스를 감지하면 사전 렌더링된 완성형 음절 글리프를 VRAM에 직접 전송하는 자체 디스패치·렌더 경로를 신설했다. 자모를 실행 시점에 합성하지는 않았다. 글리프는 완성형으로 두고 렌더 경로만 새로 깔았다.

이 단계는 PoC 비용이 본 구현 비용에 근접한다. 따라서 PoC를 더 좁게 쪼갠다 — 먼저 "신설 렌더 경로로 글리프 타일 1개를 VRAM에 올려 표시"처럼 핵심 가정 하나만 검증하는 마이크로 PoC를 만들고, 그 다음 실제 텍스트 바이트 또는 제어 흐름에서 그 경로가 호출되는지 확인한다. 전자는 렌더 경로 검증이고, 후자까지 통과해야 한글 1자 PoC다.

### 단계 선택 기준 요약

| 조건 | 단계 |
|------|------|
| 폰트가 게임 데이터 안에 있고, 글리프 크기가 한글 표현에 충분 | 1 — 글리프 교체 |
| 폰트가 BIOS 등 외부에 있거나, 인코딩 영역을 새로 정의해야 함 | 2 — fetch 훅 |
| 글리프 합성·코드 체계 신설 등 처리 경로 자체가 새로 필요 | 3 — 자체 디스패치 (PoC 단위 축소) |

## 3. PoC 검증 기준

"화면에 뭔가 나왔다"로는 부족하다. PoC 통과는 다음 세 기준을 모두 만족해야 선언한다.

### 기준 1 — 화면 출력

- 목표 한글 글자가 **의도한 위치에, 올바른 자형으로** 표시된다.
- 자형이 깨져 보이면 글리프 포맷 이해(비트플레인 배열, 행 순서, 엔디언)가 틀린 것이다. 모양이 아예 다르면 인코딩 → 글리프 매핑이 틀린 것이다. 둘은 다른 버그이므로 구분해 기록한다.

### 기준 2 — 주변 비파괴

- 같은 텍스트 박스의 다른 글자, 같은 화면의 UI, 다른 장면의 텍스트가 패치 전과 동일하게 표시된다.
- 글리프 슬롯을 잘못 잡으면 다른 글자가 함께 바뀌고, 압축 데이터를 잘못 재압축하면 무관한 그래픽이 깨진다. 패치 전 ROM과 패치 후 ROM을 같은 진행 지점에서 비교 스크린샷으로 확인한다.

### 기준 3 — 기본 동작 유지

- 부팅 → 타이틀 → 해당 장면 도달 → 세이브/로드 → 메뉴 진입·이탈 → 장면 전환이 행(hang)이나 크래시 없이 동작한다.
- 텍스트 데이터의 길이를 1바이트라도 바꿨다면 이 기준이 특히 중요하다. 제어코드 시퀀스 뒤에 잘못 끼어든 바이트는 엔진이 제어코드의 파라미터로 오인해 하드락을 일으킨다 — 새턴 마도물어 사례에서 정렬 패딩이 종단 제어코드 뒤에 삽입되어 하드락이 발생했고, 패딩을 제어코드 앞으로 옮겨 해결한 전례가 있다. PoC 단계에서는 가능하면 **원본과 동일한 바이트 길이를 유지**(고정길이 교체)해 이 변수 자체를 제거한다.

검증은 에뮬레이터에서 수행한다. VRAM 뷰어·타일 뷰어·메모리 워치가 있는 디버깅 에뮬레이터를 쓰면 "화면에 안 나올 때 어디까지는 갔는지"(글리프가 VRAM까지는 도달했는지, 타일맵 엔트리는 올바른지)를 단계별로 추적할 수 있다. 플랫폼별 권장 에뮬레이터와 디버깅 기능은 `references/platforms/` 해당 문서를 본다.

## 4. 실패 신호와 대응

PoC가 막히는 지점은 대체로 3요소 중 하나로 환원된다. 신호별 표준 대응:

### 글리프 공간 부족 → 서브셋 축소

**신호**: 필요한 글리프 수가 슬롯 상한을 넘는다. 폰트 영역의 빈 공간이 모자라거나, 하드웨어 타일 참조 한도(타일맵 엔트리의 비트 폭 등)에 걸린다.

**대응**:

- 완성형 전체(2,350자)를 넣겠다는 전제를 버린다. **번역에 실제 등장하는 글자만** 수집해 할당하는 동적 서브셋이 기본 전략이다. 새턴 마도물어 사례에서는 하드웨어 한도로 산정된 슬롯 수보다 번역 어휘의 고유 글자 수가 많아, 번역 어휘 쪽을 조정해 슬롯 안에 수용했다 — 글리프 한도는 폰트 문제가 아니라 번역 어휘 관리 문제가 된다.
- 미사용 글리프 슬롯 회수(사용되지 않는 기호·중복 문자), 숫자·영문 슬롯의 직접 재활용으로 가용 슬롯을 늘린다.
- 그래도 부족하면 자모 조합형(단계 3)으로의 전환을 검토한다. 단 이는 PoC 재설계를 의미한다.

### 인코딩 충돌 → prefix 재탐색

**신호**: 한글용으로 잡은 코드 영역이 기존 데이터와 충돌한다. 일본어 원문이 그 영역을 실제로 쓰고 있거나, 텍스트가 아닌 바이너리 값(좌표, 카운터)이 한글 코드로 오인 해석되거나, 엔진의 문자/명령 판정 분기가 해당 영역을 텍스트로 취급하지 않는다.

**대응**:

- 원본 텍스트 전량을 덤프해 lead byte 사용 분포를 만들고, **실제로 비어 있는 영역**을 다시 고른다. 추정으로 고르지 않는다. 원문 인코딩의 미정의/사용자 정의 대역(원문에 등장할 수 없는 코드 대역 — SJIS라면 사용자 정의 영역, 1바이트 테이블이라면 미사용 상위 대역)이 일반적인 1순위 후보다.
- 엔진의 텍스트/명령 판정 분기 조건을 디스어셈블리로 확인해, 새 prefix가 기존 분기를 그대로 통과하는 영역인지 검증한다. 분기를 통과하지 못하는 영역을 골랐다면 prefix를 바꾸는 쪽이 분기를 고치는 쪽보다 싸다.
- 텍스트 영역과 비텍스트 데이터가 섞여 흐르는 포맷이라면, 코드 영역 선택만으로 부족하고 문맥 검증(주변 바이트 패턴 확인)을 추가해야 할 수 있다. 새턴 마도물어 사례에서는 좌표값이 포인터 prefix로 오인되는 false positive를 suffix 패턴 검증으로 걸러냈다 — 인코딩/패턴 충돌은 "영역 재선정 + 문맥 검증" 조합으로 푼다.

### 렌더링 경로 복잡 → 훅 지점 변경

**신호**: 글리프와 코드를 다 넣었는데 화면에 나오지 않거나, 텍스트 경로가 여러 갈래(대화/메뉴/전투가 각각 다른 루틴)라 어디를 건드려야 할지 발산한다. 훅을 걸었더니 영향 범위가 예상보다 커서 무관한 표시가 깨진다.

**대응**:

- 훅 지점을 **경로의 합류점**으로 옮긴다. 상류(텍스트 인터프리터의 문자 분기)는 경로마다 따로 존재하기 쉽고, 하류(글리프 fetch 직전, VRAM 전송 직전)일수록 모든 경로가 합류한다. PCE 마도물어 1 사례에서 설계 후보를 비교한 결과, 상류의 렌더링 엔진을 통째로 바꾸는 설계는 후킹·교체 지점이 5곳 이상 필요했지만 글리프 fetch 호출 후킹은 1곳으로 끝났다.
- 어느 레이어/하드웨어 경로가 실제 사용되는지 실측으로 확정한다. 코드에 존재하는 경로와 실제 쓰이는 경로는 다르다 — 에뮬레이터 레이어 토글, 쓰기 브레이크포인트로 확인한다.
- PoC 범위를 한 경로로 좁힌다. 대화 텍스트 하나만 통과시키고, 메뉴·전투는 본 구현 과제로 넘긴다.

세 대응 모두 공통 원칙은 같다 — **PoC 단계의 막힘은 전제(슬롯 산정·코드 영역·훅 지점)를 다시 고르라는 신호다.** 구현을 정교하게 다듬을 때가 아니다.

## 5. PoC → 본 구현 승격

PoC가 통과하면 가장 흔한 유혹은 PoC 코드를 키워서 본 구현으로 만드는 길이다. 하지만 PoC 코드는 수동 오프셋, 하드코딩된 1자 매핑, 임시 헥스 편집 위에 서 있다. 이것을 키우면 부채가 본 구현 전체에 스며든다.

**원칙: 코드는 버리고 지식만 승격한다.**

승격 대상은 PoC가 *확정한 사실*이다.

| 승격하는 지식 | 예 |
|---------------|-----|
| 글리프 포맷 | 타일 크기, bpp, 비트플레인 배열, 압축 방식 |
| 폰트 위치와 한계 | 폰트 데이터 오프셋, 가용 슬롯 수, 하드웨어 타일 상한 |
| 인코딩 설계 | 한글 코드 영역, 코드 → 글리프 인덱스 변환식 |
| 훅 좌표 | 후킹 주소, 원본 명령, 입출력 규약, 보존해야 할 레지스터 |
| 검증된 경로 | 어느 하드웨어 레이어로 렌더링되는가, fetch → 변환 → 전송 흐름 |
| 함정 목록 | 하드락 유발 조건, false positive 패턴, 보존 슬롯 |

이 사실들을 분석 문서에 좌표·근거와 함께 기록한 뒤, **본 구현은 빌드 파이프라인 위에서 재작성한다.** 본 구현의 기준은 "원본 ROM + 번역 데이터 + 폰트 → 패치 ROM"을 명령 한 번으로 재생성하는 것이다. 글리프 수집·할당, 텍스트 인코딩, 포인터 수정, 재압축, 디스크 기록까지 모든 변환이 파이프라인 안에서 자동으로 일어나야 하며, PoC 시절의 수동 편집 산출물이 입력에 끼어 있으면 안 된다. 번역이 한 줄 바뀔 때마다 전체를 다시 빌드할 수 있어야 수백 파일 규모의 본 작업이 유지된다.

권장 승격 순서는 이렇다: 한글 1자 PoC(글리프·인코딩·렌더 경로 검증) → 대표 텍스트 1단위의 엔드투엔드 패치(글리프 수집·인코딩·포인터 수정·재압축까지 한 파이프라인으로 검증) → 본 구현(전체 번역 디렉토리를 일괄 패치하는 단일 빌드 커맨드). 각 단계에서 이전 단계의 임시 코드는 파이프라인의 정식 모듈로 대체하고, 살아남는 것은 글리프 포맷·포인터 패턴·제어코드 규약 같은 검증된 지식뿐이다.

대표 텍스트 1단위의 엔드투엔드 번역 패치는 1자 PoC와 본 구현 사이의 권장 중간 단계다. 1자 PoC가 렌더링 3요소를 검증한다면, 이 중간 단계는 글리프 *수집·할당의 자동화*와 *가변 길이 텍스트의 부작용*(포인터, 패딩, 크기 한도)을 검증한다. 이 둘이 모두 통과한 뒤에 전량 작업에 들어간다.
